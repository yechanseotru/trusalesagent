TruAbutment Sales Agent — Production Build

Foundation documentation for vendor architectural review and recommendation.

01Project Summary

Context for what we're building, who it's for, and the shape of the engagement.

What we're building

TruAbutment's AI Sales Agent is a conversational ordering and product Q&A assistant for dental professionals — dentists (clinicians) and dental lab specialists. This engagement is the first proper production build, moving on from an in-house prototype.

User types — both supported in this phase

Non-login user

Enters first name, phone number, email, and country at the start. Can browse, ask product Q&A, and receive pricing at MSRP.

Cannot place orders and cannot access order history. When a non-login user tries either, the bot guides them to log in, and points to sign-up if they don't yet have an account.

CP-account user (logged in)

Authenticated through the Customer Portal. Sees account-specific pricing (per-customer tiers applied), not MSRP.

Can place orders through the full order flow and access order history.

Both user types are in scope for this phase. The split affects pricing, ordering capability, and which signals fire — but the conversational experience should feel consistent across both.

Channels & Architecture

Primary — iOS app

Custom Chat API client for dental professionals. Conversational ordering and product Q&A. A .NET Proxy on our side handles auth, CP API bridging, signal routing, and TTS/STT integration; what reaches BotPress is always text in / text out.

Future — Custom Web

Same bot, same backbone, custom web client (replacing the previous embedded Webchat prototype, which is being deprecated). Single BotPress bot serving multiple custom clients.

Voice Mode

Voice mode is not on by default. The user toggles it on via a button in the iOS app, and can turn it off at any time. The system never re-enables it automatically. Voice mode turns off automatically only at the address picker step, since selection happens via native dropdowns.

Voice processing happens entirely on our side. STT (speech-to-text) and TTS (text-to-speech) are handled by our chosen providers, integrated through the .NET Proxy and the iOS app. BotPress only sees and produces plain text — what the user said is transcribed before reaching the bot, and what the bot responds with is sent through TTS on its way back to the user.

Core Principles

  • Naturally conversational tone — across voice and text. Because the bot's response text is sent directly to TTS when voice is on, and shown as a chat bubble when voice is off, the same response style needs to work for both. We're aiming for a middle-ground tone that reads naturally aloud without being terse, and reads naturally on screen without being verbose. Alternatively, if BotPress can produce two response tracks per turn (a voice-tuned variant and a chat-tuned variant) keyed off voice state, that's also acceptable. We're open to suggestions here.
  • No hallucination. Only verified facts. No invented SKUs, prices, or compatibility claims.
  • Honest redirect when uncertain. When the bot doesn't have what it needs, it says so and points to the right contact.
  • No autonomous orders or medical decisions. The user always confirms; the bot never advises clinical choices.

What we want from this engagement

  • System maintainable — production-grade, debuggable, modular.
  • KB maintainable — content can be updated by us going forward, ideally through a tooling layer rather than the studio dashboard.
  • Modular workflow architecture — clean boundaries between conversational core, order flow, and integrations.

02Catalog Reality

The current state of our catalog — what's firm, what's still being organized, and a structural quirk worth flagging.

Status

Catalog categorization is in progress on our side. For the initial launch, what we have firm KB content on is six main Product Family Kits and their internal stock components — approximately 150 items in total.

Within these 150 items, two levels exist: the kit itself (the main, sellable product family unit) and the stock-level components that make up that kit (also individually orderable). Both levels are within the initial in-cart ordering scope.

Beyond the 150, there's a broader stock catalog of several thousand SKUs across various item categories. These aren't covered by this initial scope — they'll be addressed in a subsequent scoping phase. We'll go over the structure together at that point, or share supporting materials on request.

Initial Product Family Kits

  • TruBolt Kit
  • ioConnect Kit
  • T-Marker Kit
  • Tru Reamer Kit
  • Tru Dual Align Scan Body Kit
  • ASC Pro Kit
Each kit has documented KB content and a set of internal stock components. Kit and components together amount to roughly 150 SKUs.

Item Description Compounding

Item names embed multiple dimensions of information — product family, implant brand, size, and length variant — in a single string. Examples from the initial list:

IO-ST46L ioConnect compatible with Straumann 4.6 Long HSS-GD47 HS Cap Small compatible with Global D 4.7 TRMK01-AU Tru Reamer Kit compatible with AOT & URIS ST46-DWFRM Direct Screw Final Reamer compatible with Straumann 4.6
A single name typically packs: product family · component type · implant brand · size · length variant. Users frequently search by one of these embedded dimensions ("Straumann 4.6 compatible", "Long", "AOT reamer") rather than by exact SKU.
We'd appreciate BotPress's input on rules or strategies for parsing and searching across this kind of compound-description data — what's worked in your other catalog-heavy bots.

KB Standardized Format

We've established a structural standard that all product KBs follow. The essentials are below.

Chunking rules (current working set)

Each ## section follows four rules that have been giving us good retrieval performance so far:

  1. Independent — no "as mentioned above" type references; each section reads standalone.
  2. Self-complete — retrieved in isolation, it answers its core question.
  3. Header contains searchable keywords — "ioConnect Scan Body — Sizes and Specifications", not just "Sizes".
  4. Keywords line at section end — canonical name, alternative form, itemNo, PDF description form, and likely user query phrasings.

Target length: 100–400 words per section. We're open to suggestions on how content structuring could be enhanced further.

Cross-product references — hub-spoke

Product KBs mention other products by name where relevant, but detailed interactions (combined workflows, compatibility matrices, decision criteria) live in a separate Product Relations document. The intent is to keep individual product KBs focused, keep cross-product information in one canonical place, and keep growth manageable as new products are added.

Source-of-truth hierarchy

When sources conflict, this order resolves it:

  1. PRD Target Item List — itemNo, SKU identity, descriptions, kit membership
  2. IFU documents — regulatory, safety, sterilization parameters
  3. Latest official brochure / PDF — canonical product spec, kit composition
  4. Customer policy (FAQ / support) — customer-facing policy and responses
  5. Product data (marketing) — features, technical specs, positioning
  6. Existing KB content — overwritten by any of the above when in conflict

KB Maintenance

Going forward, we want to be able to maintain KB content on our side without editing the BotPress studio directly. A KB-update tool of some form is part of the desired outcome of this engagement. We're open to BotPress's recommendation on what shape that tool should take.

Open questions for BotPress

  1. Recommended data architecture for catalog at this scale — KB documents, function tools, hybrid?
  2. Strategies for handling compound item descriptions where brand and compatibility are embedded in the name?
  3. How would retrieval design need to evolve as we expand from the initial 150 toward the broader catalog later?
  4. Shape of a KB-update tool that lets us maintain content without going into the studio?

03Order Flow Design

Initial design direction for the order flow. Open to refinement based on BotPress's experience with similar order bots.

Where the user commits

The order flow has several steps, but the overall arc is driven by two big user inputs. The first is Proceed — the user has built their cart in chat and is ready to lock in the items; this creates the order in CP. The second is Submit — after reviewing shipping, contact, and delivery details in a native picker, the user confirms the order for final submission. Cart building happens in chat (natural language); the picker happens in native UI; both are needed because each part of the flow has different ergonomics.

Conversational quality during ordering

All cart building happens through natural language, so utterance intent classification is central. While the user is in ordering mode, they may stay on-task ("add two more", "actually make it Straumann instead") or shift to adjacent questions ("what's the lead time?", "is this compatible with X?"). When the question is off the ordering track, our intent is to let the user know we're stepping out of ordering mode, route back through the main orchestration node to the relevant expert workflow for a proper answer, and then offer to return to ordering. Forcing every adjacent question to be handled inside the order workflow would either stretch its scope too far or risk shallow answers.

A note on cart state: before Step 2 (create), the cart lives only in BotPress state. If the user exits ordering mode for an extended troubleshooting or Q&A detour, cart state may be lost depending on how state preservation is handled. We'd value BotPress's view on the cleanest pattern for preserving in-progress cart state across mode exits — or whether prompting the user to commit (Proceed) before extended detours is the more practical approach.

Cart visualization

As items are added or modified, we'd like the cart preview to show item images alongside names and quantities, similar to a standard e-commerce cart. The image data will be available on our side; this is mainly a presentation note for how the cart preview signal renders on the iOS client.

Operating Rules

  1. Before Proceed, the cart lives in BotPress state. CP is never touched for order creation.
  2. After create, the order exists in CP with OPEN status and its lineItems are locked. To change items, the user starts a new order.
  3. If the user exits without confirming, the OPEN order stays as Pending — the bot never calls a cancel API.
  4. On resume, the user is offered two choices: Continue to Submit or Start New.
  5. Initial in-cart ordering covers the ~150 firm SKUs. The broader catalog isn't part of this initial ordering scope, but the structure should be scalable — eventually we'd like users to order from the broader catalog conversationally as well, which means the item-searching logic has to handle the complexity of natural language across thousands of SKUs. That's a non-trivial design problem and we'd like the architecture to accommodate it from the start.

Cart Modification Window

Item additions, quantity modifications, and deletions are freely available during Step 1 only. From Step 2 onward (after create), line items are locked. The user can still cancel and start a new order, but the existing OPEN order isn't edited.

This is our initial design direction. If BotPress sees better patterns for any part of this flow — particularly around state preservation, workflow handoff to native UI, recommendation timing, scalable item searching, or cart modification boundaries — we welcome the input and would refine accordingly.

04API Integration Scope

How the order flow's API calls are organized. All calls — whether originating from BotPress or from the iOS app — route through our .NET Proxy, which will be the single integration boundary to CP. The Proxy is in development; its dedicated documentation will be shared when ready. Endpoint shapes for the calls BotPress initiates are included in the API Reference section at the end of this document.

Single integration boundary — the .NET Proxy

From BotPress's perspective, there will be one API target: the .NET Proxy. BotPress won't call CP directly. The Proxy handles authentication, payload transformation where needed, logging, and forwarding to CP. CP schemas remain the source of truth for the underlying contracts (visible in the CP Swagger), but the actual request target BotPress hits will be the Proxy.

This keeps CP credentials and connection details contained, simplifies BotPress workflow configuration to a single base URL, and lets us absorb any future CP API changes in the Proxy layer without touching BotPress.

BotPress-side calls outside the order flow

A few calls happen outside the four order steps — mostly session setup and grounding data. All go through the Proxy.

GET /customers?customerNo={no} Session enrichment when a user signs in — Proxy fetches account context for the conversation.
GET /pricing/catalog/{customerNo} Account-aware item grounding (price tiers, available SKUs for this account).

API Sequence by Step

Step 1 and 2 are initiated by BotPress; Step 3 and 4 by iOS. Every call listed below goes through the Proxy. Request and response shapes for the BotPress-initiated calls (Step 1 estimate, Step 2 create) are in the API Reference section.

Step 1 · Cart — BotPress → Proxy → CP
POST /orders/estimate Lightweight call with customerNo + lineItems. Returns price + stock validation for the current cart state.
Step 2 · Create — BotPress → Proxy → CP
POST/orders/createMinimal payload: lineItems + type=SO + caseName. Generates the OPEN order in CP.
Step 3 · Picker Data — iOS → Proxy → CP
GET/addresses/shiptoShip-to dropdown
GET/contacts/contactContact dropdown (fills billTo)
GET/datadictionary/delivery-options?country={c}Country-filtered delivery methods
Step 4 · Finalize — iOS → Proxy → CP
POST/orders/update/{orderId}Writes the three picker selections (no lineItems)
POST/orders/submit/{orderId}Final validation. OPEN → CONFIRMED. Result signaled back to BotPress for closure messaging.

Resume & Result Handoff

Our current thinking is that BotPress wouldn't need to re-fetch order status from CP just to handle the Step 4 result. Instead:

  • On success — the submission success response is signaled back to the conversation, which delivers a closure message inline.
  • On failure — the failure detail is signaled back as well, so the user can be guided on next steps in the same conversation.

Both outcomes flow back as signals within the same session that initiated submit. For pending state across sessions, see Section 05 — our preference is for that to reset on new sessions, with CP serving as the source of truth when a user later wants to check or resume an earlier order.

Fallback paths — what happens if the signal doesn't arrive, if the session has expired by the time it does, if the user re-enters mid-finalization — would benefit from a more deliberate strategy than we've sketched. We'd value BotPress's recommendations on the right pattern here.

Authentication

The Proxy handles authentication to CP on behalf of both BotPress and iOS. BotPress only needs to identify the user context (customerNo or equivalent) in its calls to the Proxy — no CP-side credentials live in BotPress.

Error Handling Principles

  • 400 / validation: user-facing message + retry path.
  • 401 / auth: session refresh, then retry.
  • 5xx / server: graceful message + HITL email digest to internal team.
  • Network / timeout: one retry, then HITL.
  • Stock unavailable at submit: block the order, sales redirect.

Idempotency

  • estimate — safe to retry, read-only.
  • create — avoid duplicates within a session (cart-once principle).
  • submit — must never be called twice (financial-transaction safety).

Conventions

  • type is always "SO" (Stock Order)
  • caseName uses prefix TruAgent-{timestamp}
  • BotPress never updates lineItems via the update endpoint

Open questions for BotPress

  1. BotPress-native pattern for HTTP API calls (to the Proxy) — built-in HTTP nodes, custom actions, or other?
  2. Recommended idempotency lock for submit?
  3. Retry logic at workflow level vs HTTP layer?
  4. Fallback design. How should fallback paths (API failures, ambiguous user input, dormant-state expiry, mid-flow exits, missing result signals) be designed organically inside the BotPress workflow rather than bolted on? We'd value BotPress's view on building these in from the start.

05iOS Signal ContractDraft

Starting-point proposal for the structured events exchanged between the bot and iOS via the .NET Proxy. Some items below require BotPress's confirmation before we finalize.

Approach

Most conversation is plain text in / plain text out. At specific moments BotPress needs to trigger native iOS UI (login sheet, address picker, order detail view) or carry structured data (cart, order result). These moments are encoded as signals — structured events embedded in the standard message payload.

Order Finalization Split

In our current sketch, once ORDER_SUBMIT_GATE goes out, BotPress's role pauses while the Proxy handles the CP API update + submit chain. iOS then reports the final result back via ORDER_RESULT, and BotPress picks back up for closure messaging. The intent is to keep the heavier, more failure-prone part of the flow outside the conversation layer — but we'd like to hear whether this kind of pause-and-resume sits well with how BotPress workflows are typically built.

Outbound Signals — Bot → iOS

A shape we've been considering for outbound messages — not finalized. We'd value BotPress's guidance on what's most natural for BotPress communication patterns, and what we should adjust:

{ "text": "<chat bubble content; may be empty>", "payload": { "event": "<EVENT_NAME>", "data": { ... }, "voiceMode": "off" | null, "buttons": [{ "label": "...", "value": "..." }] | null, "followUpSuggestions": ["...", "..."] | null } }

Voice Mode in Signals

Voice mode lives on the iOS client side — it's not something BotPress needs to track. When voice mode is on, user speech is converted to text via STT before reaching BotPress, and BotPress responses are converted to speech via TTS on the way back. From BotPress's perspective, every turn is plain text in and plain text out, regardless of whether voice is on.

In our sketch, BotPress could optionally include voiceMode: "off" on outbound messages as a hint to the client to suppress TTS at structured moments (submit gate, order success / failure). iOS would read this as a mute override, and voice would stay off until the user re-enables manually. Whether this lives in the payload or as something else BotPress-native is open.

Images in Signals

The cart preview (and possibly some product Q&A responses) should include item images, as noted in the order flow section. We'd value BotPress's recommendation on the cleanest way to carry image data in the message payload — image URLs in the data object, separate attachment fields, or another pattern. The image data itself is available on our side; this is about how the signal references it.

Pending Order Handling

  • Within a single session: we'd see BotPress holding the pending order reference in conversation state. If the user mentions a new order intent while one's still pending, the bot can surface an inline Yes/No "discard pending and start new?" prompt.
  • Across sessions: our preference is for pending state not to carry forward. A new session starts fresh, and if the user wants to check or resume a previous order, the bot can look it up via CP at that point.
  • "Discard" semantics: clears the pending reference held in the current session. The CP-side order stays as a dummy OPEN entry — intentional, no CP cleanup call from the bot.
The cross-session reset is a deliberate simplification on our side — keeping the conversation layer light and letting CP be the source of truth for order status. If a returning user needs a "you have a pending order" surface, the iOS app can handle it via its own CP API lookup at session start. Open to BotPress's view on whether this division makes sense or whether you'd recommend something different.

Open questions for BotPress

  1. Inbound structured events. What's the ADK-native pattern for clients to send structured events (not plain text) that workflow nodes can read and branch on? Our draft assumes { type: "custom", payload: { event, data } }. Is this right? How does a workflow node read payload.event?
  2. Outbound payload structure. Can outbound BotPress messages carry arbitrary metadata fields in the payload (e.g., voiceMode, buttons, followUpSuggestions)? Or do these need to use BotPress-standard fields? In ADK, what's the canonical shape?
  3. Buttons and quick replies. How are inline buttons represented in ADK Chat API responses? Are tapped buttons sent back as plain text matching the button value, or as structured events?
  4. Follow-up suggestions array (sales tool). As a sales agent, every BotPress response is a potential moment to surface a relevant next step. We'd like BotPress to optionally return a small array (around three) of follow-up question suggestions per message — strings the iOS app can render as tappable chips, similar to how chat apps offer quick replies. The app may show all, some, or none depending on context. Is this kind of metadata array supported in the outbound payload?
  5. Transactional ordering mode — trigger and cart management. The bot needs to distinguish generic ordering interest ("how much is X?") from transactional ordering intent ("order one for me"). We'd value BotPress's view on the cleanest way to trigger transactional mode and keep cart management organically tied to it — including how the bot can switch back to generic mode and preserve cart state when needed.
  6. Conversation dormancy. After ORDER_SUBMIT_GATE, BotPress may wait 30 seconds to several minutes for ORDER_RESULT. What's the idle timeout? Configurable? What happens to conversation state if it expires?
  7. Multi-client single bot. The same BotPress bot will eventually serve iOS and a future custom web client. Any per-client patterns we should know about?
  8. Conversation state handling for in-progress orders. The cart (pre-create) and pending-order reference (post-create) live in conversation state. We'd like to understand how state like this is best handled across the different moments — across turns within a session, across a dormant window while iOS finalizes, and across sessions if the user returns later. We're less looking for raw TTL numbers and more for the right pattern given each situation.
  9. Non-login user session data injection. Non-login users enter first name, phone, and email at the start. What's the recommended way to inject this kind of custom session data into a BotPress conversation, and have it available throughout the workflow?
Where ADK constraints differ from what we've drawn, we'll adapt. The intent here is to share a starting-point proposal — what's most natural to BotPress is what we'd like to follow.

06Business Rules Summary

Key principles at a high level. Many specific rules already exist in our prototype's orchestration; this summary highlights what matters most and flags where continued discussion is expected.

Key Principles

  • Item searching, matching, and compatibility checks must be precise. Compound item descriptions (brand + size + system embedded in name) need a robust matching strategy — see Section 02.
  • Clarification gate when uncertain. When a user query is ambiguous (third-party brand mentioned without product context, vague intent, missing key parameter), the bot should ask one focused clarifying question rather than guessing.
  • Honest redirect when data unavailable. For items or topics outside the bot's reach, redirect to the appropriate contact rather than fabricating.

Ordering Scope & Scalability

The initial in-cart ordering scope covers the ~150 firm SKUs — the six Product Family Kits and their internal stock components. The broader catalog of several thousand stock items isn't covered by this initial scope, but it is firmly in the longer-term picture.

For that reason, scale-up structure needs to be reflected in the architecture from the start — not retrofitted later. The challenge with the broader catalog is the natural-language matching: items embed multiple dimensions in their names (brand, size, system, variant — see Section 02), and user phrasings around them are correspondingly varied. What we'd like from BotPress is a view on the strategy and structure for handling this at scale:

  • Is there a feasible path to in-cart ordering across the broader catalog conversationally? If yes, what would the structure look like?
  • If full-catalog ordering isn't feasible in one step, what's a reasonable intermediate scope — specific item categories that are tractable first, before expanding outward?
  • How should the initial architecture be shaped so the broader catalog can be folded in later without rebuilding the order flow?

Conversational handling of the broader catalog for Q&A and lookup is a separate matter and a positive outcome if achievable in this engagement.

Additional business rules will emerge through continued discussion as the design evolves. This summary is not exhaustive — we expect to polish these together in subsequent scoping phases. We'll surface specific existing-system context (rule details, routing logic, etc.) as BotPress asks for it during scoping.

07User Utterance Examples

A sense of what real user conversations look like. Not exhaustive — these are representative scenarios meant to ground retrieval and conversation design decisions.

User Population

Users are dental professionals — both dentists (clinicians) and dental lab specialists. They speak in clinical and technical terms specific to dental implantology, not in SKU codes. Korean and English are sometimes mixed in the same query.

A note on the bot lines below — they're written so the same wording works for both screen and voice. The screen is always on (whether voice mode is enabled or not), so phrasing has to read cleanly visually first. Items like SKU codes, prices, and emails are written in their standard form; modern TTS handles them naturally without needing them spelled out.

Pending state isn't auto-surfaced at session start. It comes up when the user mentions a prior order or starts a new order intent. CP remains the source of truth for pending status.
6 · Broader-catalog item — outside ordering scope
Do you have a TruBase abutment for Astra Tech 4.2?
TruBase is part of our broader catalog, not in the kits I can place orders for right now. For the specific variant and availability, our sales team can confirm and help directly. You can reach them at sales@truabutment.com. Want me to share that contact?
Item is outside the initial in-cart scope. The bot acknowledges, doesn't invent details, and redirects naturally.
These are illustrative. The bot lines are written to read cleanly on screen and pass through TTS naturally when voice mode is on — the same wording works for both. We can provide more examples or specific clinical term references as needed during development.

08Priority for First Response

A few questions surfaced throughout this document carry the most weight for unblocking work on our side — iOS app development, .NET Proxy architecture, and conversation tone direction all depend on these. We've grouped them here as the items we'd most value early input on, before full scoping.
1 · ADK Chat API — structural constraints

What are the structural limits of the ADK Chat API around message payloads? Specifically: can outbound messages carry arbitrary custom metadata fields (signals, buttons, follow-up suggestions, voice hints, image references) within the standard payload, or do these need to conform to specific BotPress-defined fields? Understanding this shapes how the iOS client and the Proxy parse and pass messages — we'd like to know before locking either side's structure.

2 · Event signals — fully customizable?

Section 05 sketches a signal contract (outbound BotPress → iOS events like ORDER_SUBMIT_GATE, RESUME_PROMPT, and inbound iOS → BotPress events like RESUME_DECISION, ORDER_RESULT). Can these be fully defined and routed on both directions in ADK, or are there constraints we should design around? If structured inbound events aren't well supported, the iOS app and Proxy will need a different translation pattern.

3 · Voice mode response strategy

Because BotPress responses go through TTS unmodified when voice mode is on, the response style matters a lot. Two paths we're considering:

  • Two response tracks per turn — BotPress returns both a voice-optimized version and a screen-optimized version in the payload, and the iOS client picks based on current voice state.
  • Single middle-ground tone — one response style designed to work for both screen and voice, accepting the trade-off.

Is the two-track approach feasible in ADK? If not, single middle-ground is what we'd pursue. This decision affects KB content phrasing, orchestration instructions, and the iOS rendering layer — it's worth settling early.

4 · What else should we know before Proxy & iOS dev starts?

Beyond the above, are there BotPress-side constraints, configuration requirements, or architectural realities we should bake into our Proxy and iOS design from day one — things that are harder to retrofit later? Authentication patterns, session management, multi-client considerations, rate limits, anything that would meaningfully shape how we build on our side.

The other questions raised throughout this document are also genuinely open — we'd welcome input on all of them in due course. The four above are simply the ones blocking the most upstream decisions, and where early answers would have the biggest impact on how we shape iOS and Proxy work in parallel with BotPress's build.

REFAPI Reference

Request and response shapes for the CP endpoints that BotPress will interact with through the Proxy. The Proxy will mirror these contracts closely — field names and shapes shown below are what BotPress workflows will work with. Endpoints related to Step 3 and Step 4 (ship-to, contact, delivery method selection, order finalization) are handled by the iOS client and are omitted here.

Base URL is environment-dependent and will be shared separately. Paths shown reflect the underlying CP contract (the Proxy will expose equivalent endpoints under its own base). All requests use Content-Type: application/json and a bearer token in the Authorization header.

POST /orders/estimate

Used in Step 1 (Cart). Lightweight call that validates line items against pricing and stock for a given customer. Called each time the cart changes; safe to retry, read-only.

Request
{ "customerNo": "TEST_CUSTOMER_01", "items": [ { "itemNo": "IOK-AT", "qty": 1 }, { "itemNo": "IOK-UR", "qty": 2 }, { "itemNo": "TRMK01-ST", "qty": 1 } ] }
Response — 200 OK
{ "status": "success", "processedItems": [ { "itemNo": "IOK-AT", "itemDescription": "ioConnect Kit compatible with AOT", "success": true, "reason": null, "price": 3000.00, "orderedQty": 1, "onhandQty": 24, "stockStatus": "InStock", "subtotal": 3000.00 }, { "itemNo": "IOK-UR", "itemDescription": "ioConnect Kit compatible with URIS", "success": true, "reason": null, "price": 3000.00, "orderedQty": 2, "onhandQty": 11, "stockStatus": "InStock", "subtotal": 6000.00 }, { "itemNo": "TRMK01-ST", "itemDescription": "Tru Reamer Kit compatible with Straumann", "success": true, "reason": null, "price": 890.00, "orderedQty": 1, "onhandQty": 8, "stockStatus": "InStock", "subtotal": 890.00 } ], "summarize": { "successCount": 3, "totalAmount": 9890.00 } }

POST /orders/create

Used in Step 2 (Create). Generates an OPEN order in CP from the locked-in cart. BotPress sends a minimal payload — shipTo, contact, and deliveryMethod are filled in later by iOS during finalize (Step 4).

Request
{ "customerNo": "TEST_CUSTOMER_01", "type": "SO", "caseName": "TruAgent-20260514093210", "lineItems": [ { "itemNo": "IOK-AT", "qty": 1 }, { "itemNo": "TRMK01-ST", "qty": 1 } ] }

{ "status": "success", "order": { "id": 1024753, "orderDate": "2026-05-14T09:32:11.4521", "subtotalAmount": 3890.00, "lineItems": [ { "itemCode": "IOK-AT", "success": true, "price": 3000.00, "orderedQty": 1, "onhandQty": 24, "stockStatus": "InStock", "subtotal": 3000.00 }, { "itemCode": "TRMK01-ST", "success": true, "price": 890.00, "orderedQty": 1, "onhandQty": 8, "stockStatus": "InStock", "subtotal": 890.00 } ] } }

GET /pricing/catalog/{customerNo}

Account-aware item catalog. Returns the entire item catalog available to a given customer in a single call — including the initial ~150 SKUs and the broader stock catalog (several thousand items). Each entry carries the account-specific price tier and current stock.

Because of the response size, how this is consumed in BotPress matters — likely cached at session start rather than re-fetched per query, or consulted through a lookup tool. We'd value BotPress's recommendation on the cleanest pattern for grounding item-level questions against a response of this scale.
